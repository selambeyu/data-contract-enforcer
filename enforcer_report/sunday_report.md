## Section 2 — Violation Deep-Dive

> **Most significant violation:** `week3-document-refinery-extractions.fact_confidence.range`
> **Severity:** HIGH | **Records affected:** 25,230 | **Source:** injected test via `inject_violations.py`

### 2a — What the Violation Is and Why It Matters

The Week 3 Document Refinery pipeline extracts structured facts from documents and writes them to `outputs/week3/extractions.jsonl`. Each extracted fact carries a `fact_confidence` field — a float representing how confident the LLM extraction model was about that fact. The data contract (generated by `src/contracts/generator.py`) profiles this field from the baseline corpus and sets a hard range clause:

```yaml
fact_confidence:
  type: float
  constraints:
    not_null: true
    minimum: 0.0
    maximum: 1.0
```

The injected test (`outputs/migrate/inject_violations.py`) simulates a producer-side bug — a developer changes the output scale from 0.0–1.0 to 0–100 without updating the contract or notifying downstream consumers. Every `fact_confidence` value in the 25,230 flattened fact rows is multiplied by 100 before writing to `outputs/week3/extractions_violated.jsonl`.

This is not a contrived scenario. It is one of the most common silent data bugs in ML pipelines: a confidence or probability score has its scale changed by a model update, a preprocessing refactor, or a library version bump, and the consuming pipeline has no schema enforcement to catch it.

**Check 2 — Statistical Drift**

| Check Type | `statistical_drift` |

The runner computes a z-score against the baseline mean and standard deviation stored in `schema_snapshots/baselines.json`. Even a 3σ drift threshold (which is already conservative) would catch this at approximately 3.0 vs the actual 4374.0.

The two checks are independent and complementary: the **range check** is a structural contract bound and fires even if the baseline is not yet calibrated. The **statistical drift check** fires based on distributional comparison and would catch subtle scale mutations that fall just within the hard bounds (e.g. a producer changing from 0.0–0.9 to 0.0–1.5 might not violate a 0.0–1.0 max if no single value exceeds 1.0, but the mean shift would be caught by drift).

## Section 3 — AI Contract Extension Results

The AI extensions (`src/contracts/ai_extensions.py`) run three independent checks against the AI-specific aspects of the data platform. All three check types have explicit thresholds and produce a PASS / WARN / FAIL verdict.

### 3a — Embedding Drift Detection

**What it checks:** Whether the semantic content of the current extraction corpus has shifted significantly from the baseline. If the LLM extraction model changes prompts, changes models, or the source documents change in nature, the distribution of extracted fact text will drift. This is caught before it corrupts any downstream similarity search or retrieval system.

**How it works:** The check computes TF-IDF vector representations of a random sample of 200 documents from both the baseline corpus and the current corpus, then measures the cosine distance between the two centroid vectors. A drift score of 0.0 means identical distributions; 1.0 means completely dissimilar.

| Metric | Raw Value |
|---|---|
| Drift Score | **0.0801** |
| Cosine Similarity | **0.9199** (1.0 = identical) |
| Threshold (WARN) | 0.25 |
| Threshold (FAIL) | 0.50 |
| Embedding Method | `tfidf_fallback` (OpenAI ada-002 not available in this environment) |
| Baseline Docs | 11,061 |
| Current Docs | 14,169 |
| Sample Size | 200 documents per comparison |
| **Verdict** | **PASS** |
| Triggered WARN/FAIL? | **No** |

**Interpretation:** A drift score of 0.0801 means the current corpus is 91.99% semantically similar to the baseline by TF-IDF cosine distance. The 3,108-document increase (14,169 vs 11,061) reflects the expanded flattened fact count after the lineage graph update — more node types means more facts to profile — but introduces no meaningful semantic shift. The content of the extracted facts has not changed; only the volume has grown. A drift score would only cross the 0.25 WARN threshold if the LLM began extracting materially different kinds of text — for example, switching from fact extraction to question generation, or if the source document domain changed from financial to medical.

> **Contract:** `week4-brownfield-cartographer-lineage`

### 4a — Background: What Changed and Why

The Week 4 Brownfield Cartographer produces a lineage graph of the entire codebase. When the lineage graph (`src/outputs/week4/lineage_graph.json`) was updated and the migration script (`outputs/migrate/week4_graph_to_snapshot.py`) re-ran, the resulting snapshot contained **three new node type values** that were absent from the previous snapshot:

---

### 4c — Compatibility Verdict (Bitol Taxonomy)

| Field | Value |
|---|---|
| Total Changes | 1 |
| Breaking Changes | 0 |
| Compatible Changes | 1 |
| Compatibility Verdict | **COMPATIBLE** |
| Version Bump Required? | **No** |

**Classification rule applied:** Under the Bitol schema evolution taxonomy, `ENUM_EXTENDED` (adding new values to an existing enum) is classified as **COMPATIBLE** — not breaking — under one condition: **consumers must handle unknown enum values gracefully**. If a consumer uses an exhaustive `switch` or `match` on `node_type` with a hard failure on unrecognised values, adding a new value would be a silent breaking change. The taxonomy makes this distinction explicit:

- **COMPATIBLE:** Consumer iterates over all nodes, uses `node_type` as a display label or filter, and unknown values simply pass through or are ignored.
- **BREAKING:** Consumer has a `switch(node_type) { case TABLE: ...; case MODEL: ...; default: throw Exception("unknown type"); }` — adding EXTERNAL would trigger the exception on every new node.

The `week7-enforcer` subscriber (sub-003) consumes `node_type` only as a label during BFS traversal — it is never used in control flow logic. New node types pass through without error.

### 5b — The Specific Silent Failure Mechanism

The Week 5 Event Sourcing Platform applies this filter before emitting an `ExtractionCompleted` event:

```python
if fact["confidence"] >= 0.5:
    emit_event("ExtractionCompleted", fact)
```

This filter is **scale-sensitive**. It assumes the confidence is in the 0.0–1.0 range, where 0.5 is the midpoint. On clean data, approximately 50% of facts pass this filter.

**What happens when `fact_confidence` changes scale from 0.0–1.0 to 0–100:**

| Condition | Clean Data | After Scale Mutation |
|---|---|---|
| Filter threshold | 0.5 | 0.5 (unchanged) |
| Fact confidence values | 0.65–0.90 | 65.0–90.0 |
| Facts passing filter | ~50% | **100%** |
| Events emitted per document | ~500 | **~1,000** |
| Event bus volume | Normal | **Doubled** |
| Any exception raised? | — | **No** |
| Any error logged? | — | **No** |
| Any check fails? | — | **No** (event schema is structurally valid) |

Every emitted event is structurally valid — it has a correct `event_id`, a valid `event_type` from the registered enum, a UUID `aggregate_id`, a monotonically increasing `sequence_number`, and well-formed timestamps. The event schema contract passes all checks. The only signal that something is wrong is the **event volume anomaly** — and that metric is not currently a contract-enforced check.

### 5d — The Second Silent Failure Mode (Compounding Risk)

There is a compounding risk that makes this interface even more dangerous than the volume explosion alone.

The `week7-ai-extensions` subscriber (sub-006) computes its embedding drift baseline using only facts with `confidence >= 0.8`:

```python
baseline_facts = [f for f in facts if f["confidence"] >= 0.8]
```

After scale mutation, this filter selects facts where `confidence >= 0.8` — which, in the 65–90 range, is **all of them** (since 65 > 0.8). The baseline is now computed from 100% of facts instead of the intended ~20% high-confidence subset. The baseline is statistically valid by construction — it will produce a low drift score on the next run — but it is computed from the wrong population. The embedding drift check then becomes a lagging indicator: it will report PASS for several runs before the distributional shift is large enough to exceed the 0.25 threshold, by which time the corrupted baseline is already embedded in `schema_snapshots/baselines.json`.

This is a **baseline poisoning** scenario. The contract enforcement system's own calibration data becomes corrupted, and subsequent drift checks are measured against a polluted reference point.

---

### 5e — What Would Fix It

**Immediate (no code change):** Both sub-001 and sub-002 are already in ENFORCE mode. As long as the ValidationRunner is integrated into the Week 3 producer's CI pipeline (i.e. runs before any downstream pipeline is triggered), the ENFORCE gate will block the corruption at the source. This architecture already exists in the codebase — it just needs to be wired into the deployment trigger.

**Short-term (one contract change):** Add an `event_volume_per_aggregate` statistical check to the Week 5 contract. The baseline would record a mean of ~500 events per document session; a 2× increase would trigger a HIGH violation even if every individual event is structurally valid. This makes the Week 5 pipeline a second independent gate.

**Structural (scale-invariant filter):** Replace the raw float threshold with a normalised comparison that reads the contract-specified `minimum` and `maximum` bounds:

```python
# Instead of:
if fact["confidence"] >= 0.5:

# Use:
norm = (fact["confidence"] - contract.minimum) / (contract.maximum - contract.minimum)
if norm >= 0.5:
```

This makes the filter scale-invariant. Whether the producer outputs 0.75 or 75.0, the normalised value is 0.75 and the filter behaves identically. It also makes the dependency on the contract explicit in code — any IDE or linter can surface it as a contract-coupled field.
